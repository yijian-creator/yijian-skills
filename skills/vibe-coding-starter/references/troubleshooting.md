# Vibe Coding 问题排查指南

> 作者：yijian｜当 AI 生成的代码出了问题，按这个指南一步步排查。

---

## 快速诊断：先判断问题类型

```
代码跑不起来？
├─ 有报错信息 → 去「A. 报错处理」
└─ 没有报错但效果不对 → 去「B. 效果不符预期」

AI 改坏了其他功能？→ 去「C. AI 引入了新 Bug」

AI 没有按要求做？→ 去「D. AI 执行偏差」

不知道从哪里开始排查？→ 去「E. 通用排查流程」
```

---

## A. 报错处理

### 第一步：完整复制错误信息

不要只截图，要把完整的错误文字复制给 AI：

```
我遇到了这个错误，请帮我分析原因并修复：

[粘贴完整错误信息，包括错误类型、行号、堆栈信息]

我的操作步骤是：[描述你做了什么触发了这个错误]
```

### 第二步：常见错误类型速查

| 错误关键词 | 可能原因 | 快速处理 |
|-----------|---------|---------|
| `Cannot find module` | 依赖没安装 | 运行 `npm install` 或 `pip install -r requirements.txt` |
| `is not defined` | 变量/函数未声明或拼写错误 | 检查变量名是否正确 |
| `Cannot read property of undefined` | 数据还没加载就使用了 | 加空值检查 |
| `CORS error` | 跨域问题 | 需要后端配置或使用代理 |
| `404 Not Found` | 接口路径错误或服务没启动 | 检查 URL 和服务状态 |
| `SyntaxError` | 代码语法错误 | 检查括号、引号是否匹配 |
| `Port already in use` | 端口被占用 | 换一个端口或关闭占用进程 |

### 第三步：如果 AI 修复后还是报错

```
修复后还是有问题，新的错误信息是：

[粘贴新的错误信息]

请重新分析，不要在上一个方案的基础上继续修改，
先回退到修改前的状态，再想新的方案。
```

## B. 效果不符预期

### 用截图 + 文字描述问题

```
代码能运行，但效果不对。

[附上截图]

期望效果：[描述应该是什么样子]
实际效果：[描述现在是什么样子]
差异点：[具体哪里不一样]
```

### 效果类问题排查清单

- [ ] 是样式问题（位置、颜色、大小不对）？→ 让 AI 检查 CSS/Tailwind 类名
- [ ] 是数据问题（显示的数据不对）？→ 让 AI 加 console.log 打印中间数据
- [ ] 是交互问题（点击没反应、事件没触发）？→ 让 AI 加调试日志
- [ ] 是时序问题（数据还没加载就渲染了）？→ 检查异步处理逻辑

## C. AI 引入了新 Bug

### 用 git diff 定位问题

```bash
# 查看最近一次提交改了什么
git diff HEAD~1 HEAD | cat

# 对比某个文件的变化
git diff HEAD~1 HEAD -- src/app.js | cat

# 查看提交历史，找到出问题前的最后一个好版本
git log --oneline | cat
```

### 把定位结果告诉 AI

```
我用 git diff 发现，AI 在上次修改中改动了这些地方：

[粘贴 git diff 的输出]

问题出在 [具体哪一行/哪个改动]，因为 [为什么这个改动有问题]。

请只回退这一处改动，不要动其他地方。
```

### 回退单个文件

```bash
# 把某个文件回退到上一个提交的版本
git checkout HEAD~1 -- src/app.js

# 或者回退到指定提交
git checkout <commit-hash> -- src/app.js
```

### 预防措施：以后每次大改前先提交

```
在开始这次修改之前，请先帮我提交当前状态：
git add -A && git commit -m "checkpoint: 修改前的稳定版本"

然后再开始修改。
```

## D. AI 执行偏差

### 问题 1：AI 只做了部分要求

**现象**：你列了 5 个要求，AI 只做了 3 个。

**处理方式**：
```
你没有完成所有要求。我列出的要求是：
1. [要求 1] ✅ 已完成
2. [要求 2] ✅ 已完成
3. [要求 3] ❌ 没有完成
4. [要求 4] ❌ 没有完成
5. [要求 5] ❌ 没有完成

请继续完成第 3、4、5 条，不要改动已完成的部分。
```

**预防方式**：在 `CLAUDE.md` 中加入"禁止偷懒"规则（见 prompts.md 中的项目规则文件模板）。

---

### 问题 2：AI 改了不该改的地方

**现象**：你让 AI 改 A，它把 B 也改了，导致 B 出问题。

**处理方式**：
```
你改动了不该改的地方。我只要求修改 [A]，
但你同时修改了 [B]，导致 [B 出现了什么问题]。

请：
1. 把 [B] 恢复到修改前的状态
2. 只保留对 [A] 的修改
```

**预防方式**：每次提需求时加上"只改这个功能相关的代码，不要动其他部分"。

---

### 问题 3：AI 理解偏差，做了不是你要的东西

**现象**：AI 做完了，但和你想要的完全不一样。

**处理方式**：
```
不对，我说的不是这个意思。

我要的是：[重新描述，尽量具体]
你做的是：[描述 AI 做了什么]
区别在于：[说明核心差异]

请撤销刚才的修改，重新理解需求后再做。
```

**预防方式**：提需求时加上"改之前先告诉我你打算怎么改，我确认后再动手"。

---

### 问题 4：长对话中 AI 忘记了之前的约定

**现象**：对话进行了很长时间后，AI 开始违反之前说好的规则。

**处理方式**：开启新对话，在开头重新说明关键约定，或者把约定写入 `CLAUDE.md` 让它自动加载。

```
我们重新开始。以下是这个项目的关键约定：
[列出最重要的 3-5 条规则]

当前任务是：[描述任务]
```

## E. 通用排查流程

当你不知道问题出在哪里时，按这个流程走：

### 第一步：二分法定位

```
最近 AI 做了多处修改，现在出问题了。
请帮我用二分法定位：
1. 先用 git log 找到最后一个正常的版本
2. 逐个还原最近的改动，确认是哪一处导致的
3. 针对性修复
```

### 第二步：隔离法

```
我怀疑是 [模块 A] 和 [模块 B] 的交互出了问题。
请帮我：
1. 先单独测试模块 A，确认它单独运行是否正常
2. 再单独测试模块 B
3. 然后测试两者组合
```

### 第三步：最小复现

```
我遇到了一个难以描述的问题。
最简单的复现步骤是：
1. [第一步]
2. [第二步]
3. 然后就出现了 [问题现象]

[附上截图]

请分析可能的根本原因。
```

## F. 常见小白误区

| 误区 | 正确做法 |
|------|---------|
| 看到报错就慌，不知道怎么办 | 把完整错误信息复制给 AI，它能看懂 |
| AI 改完就信了，没有验证 | 每次改完都要亲自运行测试 |
| 一次提很多需求，出问题不知道是哪里 | 每次只做一个功能，做完验证再继续 |
| 出问题就重新开始，不用 git | 养成 git 提交习惯，出问题可以精准回退 |
| 截图给 AI 看不清楚 | 截图要清晰，同时用文字说明问题在哪里 |
| AI 说"已修复"就相信了 | 自己运行验证，不要只看 AI 的说法 |
