# Vibe Coding 10 条核心规则

> 作者：yijian｜从实战中提炼的规则，按重要性排序。教学时优先讲前 3 条，其余按需引用。

---

## 规则 1：PRD 先行，越具体越好

**一句话**：动手前先把需求写清楚，AI 的输出质量 = 你的输入质量。

**❌ 错误示范**：
> "帮我做一个旅行 App"

**✅ 正确示范**：
> "做一个 Web 端的旅行目的地选择器：用户通过摄像头手势控制 3D 地球仪旋转，握拳停止后选定城市，调用通义千问生成旅行攻略。技术栈用 React + Three.js + MediaPipe Hands。"

**PRD 需要包含的要素**：
- 产品定位和目标用户是谁
- 核心功能列表和交互流程
- 数据结构和接口格式（如果有）
- 页面布局描述（哪怕是文字）
- 技术栈和约束条件

**给小白的简化版**：至少说清楚这 3 件事：
1. 这个东西是给谁用的？
2. 最核心的功能是什么？
3. 用什么技术做？（不知道就让 AI 推荐）

## 规则 2：小步快跑，频繁验证

**工作节奏**：
```
描述 1-2 个功能 → AI 生成代码 → 立即运行验证 → 发现问题 → 描述问题 → AI 修复 → Git 提交 → 下一个功能
```

**❌ 错误做法**：一次性描述 10 个功能，让 AI 全部实现后再测试。
**✅ 正确做法**：每次只做 1-2 个功能点，确认无误后再继续。

**为什么**：
- 问题越早发现，修复成本越低
- 小步提交让你随时可以回退
- AI 在长对话中容易"忘记"之前的约定

## 规则 3：你是审查员，不是甩手掌柜

**核心心态**：AI 是你的初级工程师，你是 Tech Lead。代码最终的质量由你负责。

AI 生成的代码常见问题：

| 问题类型 | 表现 | 你需要做的 |
|---------|------|-----------|
| **只考虑正常流程** | 没有错误处理、没有边界条件 | 主动提出异常场景："快速切换会怎样？""网络断了呢？" |
| **阈值不准确** | 灵敏度不对、动画时长不自然 | 亲自测试，告诉 AI 哪里不对 |
| **上下文丢失** | 改 A 的时候把 B 改坏了 | 每次改动后检查相关功能是否还正常 |
| **过度工程** | 简单问题用复杂方案 | 要求简化："用最简单的方式实现" |
| **幻觉** | 编造不存在的 API 或配置项 | 查文档验证，不要盲信 |

## 规则 4：善用调试模式

对于复杂的交互逻辑，设计一个调试开关：

```typescript
// 一个布尔开关，上线前关闭
const DEBUG_MODE = false;

if (DEBUG_MODE) {
  console.log('当前状态:', state);
  return; // 只打印，不触发实际动作
}
// 正常业务逻辑
```

**好处**：快速定位问题是"识别不准"还是"逻辑错误"。

## 规则 5：版本管理是生命线

**Git 提交频率**：每完成一个功能点就提交，不要攒着。

**提交信息规范**：
```
feat: 新功能描述
fix: 修复了什么问题
refactor: 重构了什么模块
docs: 文档更新
```

**关键命令**：
```bash
git diff <commit>                          # 对比两个版本的差异
git checkout <commit> -- <file>            # 回退单个文件到指定版本
git log --oneline | cat                    # 快速浏览提交历史
git diff <good_commit> HEAD -- <file>      # 对比某个文件的变化
```

**⚠️ 大段修改必须以 Git 版本为基础**：
```bash
git show HEAD:src/app.js > /tmp/base.js   # 导出干净版本
# 在 /tmp/base.js 上做修改，确认无误后写入
```

## 规则 6：约束 AI 的执行行为

在项目规则文件（`CLAUDE.md`、`.cursorrules` 等）中明确写下：

| 约束 | 说明 |
|------|------|
| **逐条拆解** | 收到指令后，先列出所有要求点，逐一确认是否全部执行 |
| **禁止偷懒** | 不得因为"改动量大"就跳过某些要求 |
| **禁止替换** | 要求改 A，不能改成 B 代替 |
| **改动验证** | 每次修改后用 grep 验证改动是否真正写入文件 |
| **回滚意识** | 改错了立刻 git checkout 回滚 |

**实战教训**：要求 AI 修改 5 处样式，它只改了 3 处，理由是"另外 2 处影响不大"。加上"禁止偷懒"规则后，这类问题基本消失。

---

## 规则 7：先读后改，禁止盲改

**核心原则**：AI 修改任何文件之前，必须先读取文件当前内容。

- **小改动**：用精确替换，只改目标代码，不影响其他部分
- **大段修改**：以 Git 版本为基础（见规则 5）
- **修改后验证**：用 grep 确认关键代码仍然存在

---

## 规则 8：代码改动必须同步更新文档

| 改动类型 | 需要更新的文档 |
|---------|---------------|
| 新增功能 | PRD 功能列表 + 技术方案 |
| 样式变更 | PRD 视觉规范 |
| 数据结构变更 | 技术方案数据结构章节 |
| Bug 修复 | 版本历史记录 |

**为什么**：Vibe Coding 迭代速度很快，代码改了 10 版，文档还停在第 1 版，后续维护会非常痛苦。

---

## 规则 9：给 AI 看截图，比文字描述更高效

**❌ 低效**：
> "页面上第三个模块的标题好像重复了，内容也有点乱"

**✅ 高效**：
> [截图] "这里标题重复了，两个城市的内容混在一起了"

截图 + 一句话 = 最高效的 Bug 报告。

## 规则 10：Prompt 也是代码

如果产品涉及大模型调用，Prompt 需要像代码一样管理：

| 维度 | 做法 |
|------|------|
| **版本管理** | Prompt 和代码一起 Git 提交 |
| **测试验证** | 改完 prompt 后调用 API 测试，不要凭感觉 |
| **场景区分** | 不同场景用不同 prompt 模板 |
| **配示例** | Few-shot（给完整示例）比纯指令稳定得多 |
| **约束明确** | "严格按照以下 5 个章节输出，不要添加额外章节" |
| **智能跳过** | "如果某项信息不适用，请直接跳过，不要输出无意义内容" |

## 规则 11：UI 设计要避免"AI 味"

**核心问题**：AI 生成的 UI 有高度雷同的视觉特征，被称为"AI Slop"，一眼就能看出来是机器生成的。

**❌ 必须避免的 4 个 AI 味特征**：

| 特征 | 表现 | 为什么要避免 |
|------|------|------------|
| **过度居中布局** | 所有内容都堆在页面正中间 | 显得呆板，缺乏设计感 |
| **紫色渐变** | 大面积紫色/蓝紫渐变背景 | AI 的"默认审美"，千篇一律 |
| **统一圆角** | 所有卡片、按钮都是相同的大圆角 | 缺乏层次感和设计变化 |
| **Inter 字体** | 清一色使用 Inter 作为正文字体 | AI 的默认选择，毫无个性 |

**✅ 正确做法**：

- **布局**：用左对齐、网格布局、非对称排版，而不是全部居中
- **配色**：根据产品调性选色，可以参考真实产品的配色方案
- **字体大小层级**：建立清晰的字体大小体系，区分标题/正文/辅助信息
  ```
  主标题：text-2xl～text-4xl（24px～36px）
  副标题：text-lg～text-xl（18px～20px）
  正文：text-sm～text-base（14px～16px）
  辅助文字：text-xs（12px），不要低于 12px
  ```
- **字体选择**：根据风格选择合适字体，中文项目可用系统字体栈，避免无脑 Inter
- **圆角**：大容器用小圆角（rounded-lg），小元素如按钮可用大圆角（rounded-full），保持变化

**给 AI 的指令模板**：
```
设计风格要求：
- 不要使用居中布局，改用左对齐或网格布局
- 不要用紫色渐变，配色参考 [你喜欢的产品，如 Linear / Notion / Vercel]
- 字体大小要有明确层级：标题 24px+，正文 14-16px，辅助信息 12px
- 不要用 Inter 字体，中文界面用系统字体
```

**实战教训**：直接让 AI 做 UI，十有八九会出现紫色渐变 + 居中布局 + 大圆角的组合。在 PRD 里提前写明设计约束，能显著提升 UI 质量。
