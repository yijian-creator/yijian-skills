## 一、项目启动模板

### 模板 1：Web 应用启动（最常用）

```
我想做一个 [产品名称]。

**目标用户**：[谁会用这个产品，比如：上班族、学生、设计师]

**核心功能**（按优先级排列）：
1. [最重要的功能，一句话描述]
2. [第二重要的功能]
3. [第三重要的功能]

**交互流程**：
用户打开页面 → [第一步操作] → [第二步操作] → [看到什么结果]

**技术要求**：
- 前端：[React / Vue / 纯 HTML+JS，不确定就写"请你推荐"]
- 样式：[Tailwind CSS / 普通 CSS，不确定就写"请你推荐"]
- 后端：[需要 / 不需要，需要的话用什么语言]
- 数据存储：[需要数据库吗？用什么？]

**约束条件**：
- [比如：不要用付费 API / 必须支持移动端 / 代码要有注释]

请先帮我梳理技术方案，确认后再开始写代码。
```

### 模板 2：脚本工具启动

```
我需要一个 [语言，如 Python] 脚本，用来 [做什么事情]。

**输入**：[脚本接收什么数据，比如：一个 CSV 文件 / 命令行参数]
**处理逻辑**：[做什么操作，比如：过滤掉空行，统计每列的平均值]
**输出**：[输出什么，比如：打印到终端 / 生成新的 CSV 文件]

**示例**：
- 输入：[举一个具体的输入例子]
- 期望输出：[对应的期望输出]

请先确认你理解了需求，然后写代码。
```

---

### 模板 3：在现有项目中新增功能

```
我有一个 [项目描述] 的项目，现在需要新增一个功能：[功能描述]。

**当前状态**：[现在项目能做什么]
**新功能要求**：
- [具体要求 1]
- [具体要求 2]

**不要改动的部分**：[哪些已有功能不能动]

请先读取相关文件，了解现有代码结构，再告诉我你的修改方案，我确认后再动手。
```

## 二、需求细化模板

当你有了大概想法但还不够具体时，用这个模板让 AI 帮你细化：

```
我想做 [一句话描述你的想法]，但我还不太清楚具体怎么做。

请帮我：
1. 把这个想法拆解成具体的功能列表
2. 推荐合适的技术栈（我的编程基础是 [零基础/懂一点/有经验]）
3. 评估大概的开发难度和时间
4. 指出可能遇到的主要挑战

然后我们再决定从哪里开始。
```

## 三、逐步开发模板

每次只做一个功能时使用：

```
现在我们来实现第 [N] 个功能：[功能名称]。

**具体要求**：
- [要求 1]
- [要求 2]

**验收标准**：做完后，我应该能看到 [什么效果] / 能做到 [什么操作]。

**注意**：
- 只改这个功能相关的代码，不要动其他部分
- 改完后告诉我需要运行什么命令来验证

改之前先告诉我你打算怎么改，我确认后再动手。
```

## 四、Bug 修复模板

遇到问题时使用：

```
我遇到了一个问题：[描述问题现象]

**复现步骤**：
1. [第一步操作]
2. [第二步操作]
3. [看到了什么错误]

**期望行为**：[应该是什么结果]

**错误信息**（如果有）：
[粘贴完整的错误信息]

[如果有截图，附上截图]

请先分析可能的原因，再告诉我修复方案，我确认后再动手。
```

## 五、代码审查模板

让 AI 帮你检查代码质量：

```
请帮我审查这段代码：[粘贴代码或说明文件路径]

重点检查：
1. 有没有明显的 Bug 或边界条件没处理
2. 有没有安全问题（比如 SQL 注入、XSS 等）
3. 性能上有没有明显的问题
4. 代码可读性如何，有没有需要加注释的地方

请按问题严重程度排序，先说最重要的。
```

## 六、项目规则文件模板（CLAUDE.md）

在项目根目录创建 `CLAUDE.md`，约束 AI 的行为：

```markdown
# 项目规则

## 技术栈
- 前端：[React 18 + TypeScript + Tailwind CSS]
- 后端：[Node.js + Express]
- 数据库：[PostgreSQL]

## 代码规范
- 使用 TypeScript，不要用 any 类型
- 用 Tailwind CSS，不要写内联样式
- 函数要有 JSDoc 注释
- 错误处理不能省略

## AI 行为约束
- 修改文件前必须先读取文件当前内容
- 收到多条指令时，逐条列出后逐一执行，不得跳过
- 不得因为"改动量大"就简化或跳过某些要求
- 要求改 A 就改 A，不能用 B 代替
- 每次修改后用 grep 验证关键代码是否正确写入
- 改错了立刻 git checkout 回滚，不在错误基础上继续修改

## Git 规范
- 每完成一个功能点就提交
- 提交信息格式：feat/fix/refactor/docs: 描述
- 大段修改必须以 git show HEAD:文件名 导出的版本为基础

## 文档同步
- 新增功能后同步更新 PRD 和技术方案文档
- 实质性改动需递增版本号
```

## 七、常用追加指令

这些短句可以追加到任何 Prompt 后面：

| 场景 | 追加指令 |
|------|---------|
| 想先看方案再动手 | "改之前先告诉我你打算怎么改，我确认后再动手。" |
| 防止 AI 改坏其他地方 | "只改这个功能相关的代码，不要动其他部分。" |
| 要求最简实现 | "用最简单的方式实现，不要过度设计。" |
| 分步执行 | "先做第一步，做完给我看，我确认后再做第二步。" |
| 要求解释 | "改完后解释一下你改了什么，为什么这样改。" |
| 防止偷懒 | "我列出的每一条要求都必须实现，不得跳过。" |
